[ single-player recipes on a shelf ]

A private shelf for the scripts your agents keep re-deriving.

pantry is a small Cloudflare Worker and D1 store for your own recipes first. Your agent can list, get, run, and push saved code instead of re-reasoning the same pattern. pantry stores the code and hands it back when asked. It never runs it.

The recipe shape

A recipe is a named JavaScript function body with an input schema, capability tags, and a private owner. pantry keeps it in D1 and returns the code verbatim when asked.

Cheap discovery

GET /recipes lists your names, descriptions, schemas, and capabilities without the code, so a model can decide a recipe fits before transferring the script.

It never runs your code

pantry stores and transports. Running a fetched recipe, and in what isolate, is the caller's trust decision. The example runner is a convenience, not a sandbox.

Why it helps

A recurring task becomes pantry.get(name) plus a local run. A model can re-derive a procedure every time and may be wrong; pantry hands back exact saved code. The defensible win is determinism and correctness: less re-derivation risk, and often fewer generated output tokens for repeated procedures.

Pantry improves determinism and avoids regenerating saved code; it can reduce output tokens, but total-token savings require sufficiently large or repeated procedures and are not broadly demonstrated. In a small exploratory my-ax run (n=1, prod Kimi K2.7), reuse reduced output tokens but raised total tokens because the tiny procedure paid input/tool overhead. Read the proof.

How a recipe gets used

discover    GET /recipes        your names + schemas, no code
fetch       GET /recipe/:name   the full recipe, including code
decide      caller's choice     run it, or re-reason it
run         caller's isolate    pantry is not involved

A saved recipe helps only for a recurring pattern. Novel work still needs reasoning, because there is no saved recipe to reuse.

[ opt-in shared pantry ]

The same shelf can be shared

Private is the default. The same interface can also read opt-in shared recipes; it is a capability, not a claim of an active network. A recipe already has an owner, so sharing relaxes one boundary: a recipe stays private unless you push it with visibility: "shared" or pantry push --shared. 

push        visibility: "shared"   publish your own recipe
list        GET /recipes?scope=shared   shared names + schemas, no code
fetch       GET /recipe/:name       your recipe wins, then shared
trust       author owner            provenance on the shared row

Writes stay owner-scoped. Shared reads show the author owner as provenance, because the server still never runs a recipe and capabilities are tags, not grants. The caller still decides whether to fetch code, whether to run it, and what isolate deserves the risk.
